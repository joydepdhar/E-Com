# Frontend Integration Guide

## 1. Project Overview

This backend is a Django + Django REST Framework application with versioned API endpoints and JWT authentication.

- **Authentication method**
  - JWT authentication using `rest_framework_simplejwt`
  - Access and refresh tokens are issued via login
  - Protected endpoints require `Authorization: Bearer <access_token>`
- **API version**
  - Primary versioned API base: `/api/v1/`
  - Legacy compatibility routes also exist at `/api/user_app/` and `/api/store/`
- **Base URL**
  - Example local backend base: `http://localhost:8000`
  - API root: `http://localhost:8000/`
  - Versioned API base: `http://localhost:8000/api/v1/`
- **Technologies**
  - Backend: Django 5, Django REST Framework 3.16, Django Filters, Simple JWT, Cloudinary
  - No React frontend source exists in this repository
  - CORS is configured for frontend origin `http://localhost:3000` in development
- **Response format**
  - Standard REST responses return plain JSON objects or paginated wrappers
  - Paginated collection responses follow DRF paginator structure
- **Error format**
  - All API errors are returned as:
    {
      "success": false,
      "message": "Validation error.",
      "errors": { ... },
      "status_code": 400
    }

## 2. Authentication

### Login endpoint
- `POST /api/user_app/login/`
- Accepts:
  - `username`
  - `password`
- Returns:
  - `access`
  - `refresh`
  - `user` object
  - `role`

### Register endpoint
- `POST /api/user_app/register/`
- Accepts:
  - `username`
  - `email`
  - `password`
  - `password2`
  - optional: `address`, `phone`, `profile_picture`
- Creates a customer account
- Returns created user data

### Refresh Token endpoint
- `POST /api/user_app/token/refresh/`
- Accepts:
  - `refresh`
- Returns:
  - new `access` token

### Logout
- No backend logout endpoint exists
- Frontend must perform logout by discarding tokens locally

### Required headers
- `Authorization: Bearer <access_token>` for protected requests
- `Content-Type: application/json` for JSON request bodies
- `Accept: application/json`
- Optional: `X-Request-ID` for request tracing

### JWT flow
1. `POST /api/user_app/login/` with credentials
2. Receive `access` + `refresh`
3. Use `access` in `Authorization` header for protected requests
4. When access expires, refresh it with `POST /api/user_app/token/refresh/`
5. On logout, delete tokens in frontend state/storage

### Token lifetime
- The backend does not explicitly configure token lifetimes
- It uses `rest_framework_simplejwt` defaults
- Typical defaults:
  - access token: ~5 minutes
  - refresh token: ~1 day
- Frontend should treat access tokens as short-lived and refresh regularly

### Protected routes
- All cart, order, shipping, payment, review, and customer dashboard endpoints
- Admin/staff dashboards and admin APIs
- Profile endpoints

### Admin routes
- `GET /api/store/admin/dashboard/`
- `GET /api/store/staff/dashboard/`
- `GET /api/store/customer/dashboard/`
- `GET /api/store/admin/orders/`
- `GET /api/store/admin/orders/<order_id>/`
- `PATCH /api/store/admin/orders/<order_id>/`
- `GET /api/store/admin/users/`
- `POST /api/store/admin/users/`
- `GET|PUT|DELETE /api/store/admin/users/<user_id>/`
- `GET /api/store/admin/customers/`
- `GET|POST /api/store/admin/products/`
- `GET|PUT|DELETE /api/store/admin/products/<product_id>/`
- `GET|PUT /api/store/admin/settings/`
- Admin routes require `admin` role, except product management routes which allow staff or admin

### Customer routes
- Cart: `/api/store/cart/`, `/api/store/cart/remove/<item_id>/`
- Orders: `/api/store/orders/`, `/api/store/orders/create/`
- Shipping: `/api/store/orders/<order_id>/shipping/`
- Payment: `/api/store/orders/<order_id>/payment/`
  - Frontend may submit `payment_method` as `Cash on Delivery`, `COD`, or `Sandbox`
  - `payment_id` is optional for COD and sandbox/test payment flows
- Reviews: `/api/store/products/<product_id>/reviews/`
- Customer dashboard: `/api/store/customer/dashboard/`

### Payment endpoints

#### Create payment
- `POST /api/store/orders/<order_id>/payment/`
- Auth: yes
- Permission: authenticated
- Request examples:
  - Default COD (preferred):
    {
      "payment_method": "Cash on Delivery"
    }
  - Explicit COD:
    {
      "payment_method": "COD"
    }
  - Gateway payment:
    {
      "payment_method": "TestGateway",
      "payment_id": "provider-reference-1"
    }
- Response:
  - On first create:
    {
      "message": "Payment pending verification",
      "payment_id": 1,
      "status": "Pending"
    }
  - On repeated request:
    {
      "message": "Payment pending verification",
      "payment_id": 1,
      "status": "Pending"
    }
- Notes:
  - Amount is derived from `order.total_price`
  - Existing payment reuses the pending payment
  - If `payment_method` is omitted, Cash on Delivery is assumed
  - If `payment_method` is COD / Cash on Delivery, `payment_id` is generated automatically
- Errors:
  - `404` order not found
  - `400` missing `payment_id` for non-COD payment methods
  - `401` unauthorized

#### Payment verification
- No backend verification endpoint exists
- Payment remains `Pending` until backend is extended
- Frontend should treat this as initial payment submission only

#### Payment status
- Status values:
  - `Pending`
  - `Successful`
  - `Failed`
- Frontend should read nested `payment.status` from the order object

#### Retry payment
- Reposting the same order payment endpoint returns existing pending payment
- There is no dedicated retry or webhook endpoint
- For multiple attempts, frontend can retry `POST /api/store/orders/<order_id>/payment/` again

## 4. Products

### List Products
- `GET /api/store/products/`
- Returns only active products in public list
- Supports pagination, search, filtering, ordering

### Product Detail
- `GET /api/store/products/<id>/`
- Returns one product record with category and image URL

### Categories
- `GET /api/store/categories/`
- Returns paginated categories
- Each item:
  - `id`, `name`, `slug`

### Search
- Supported on products by:
  - `name`
  - `description`
- Use query parameter `search=<term>`

### Pagination
- Supported by `page` and `page_size`
- Default page size: 20
- Max page size: 100

### Filtering
- Supported on product list:
  - `category`
  - `is_active`
  - `min_price`
  - `max_price`

### Sorting
- Supported on product list:
  - `name`
  - `price`
  - `created_at`
- Use `ordering=name`, `ordering=-price`, etc.

### Images
- Product `image` field returns:
  - full Cloudinary URL for Cloudinary-hosted images
  - absolute URL if running locally
- Always use `image` field from response; do not reconstruct image paths manually

### Example request
GET /api/store/products/?category=1&search=keyboard&min_price=20&max_price=100&ordering=-price&page=1&page_size=20

### Example response
{
  "count": 1,
  "next": null,
  "previous": null,
  "results": [
    {
      "id": 5,
      "name": "Wireless Keyboard",
      "description": "Compact wireless keyboard",
      "price": "49.99",
      "image": "https://res.cloudinary.com/...",
      "stock": 10,
      "is_active": true,
      "created_at": "2026-07-20T12:00:00Z",
      "category": {
        "id": 1,
        "name": "Keyboards",
        "slug": "keyboards"

## 6. Checkout

### Full checkout flow
1. Add cart items with `POST /api/store/cart/`
2. Create order from cart:
   - `POST /api/store/orders/create/`
3. Add shipping address:
   - `POST /api/store/orders/<order_id>/shipping/`
4. Make payment:
   - `POST /api/store/orders/<order_id>/payment/`

### Required request body
- `POST /api/store/orders/create/`
  - no body required
- `POST /api/store/orders/<order_id>/shipping/`
  {
    "address": "123 Main St",
    "city": "City",
    "postal_code": "10001",
    "country": "USA"
  }
- `POST /api/store/orders/<order_id>/payment/`
  {
    "payment_method": "TestGateway",
    "payment_id": "txn-123"
  }

### Validation
- Order creation validates:
  - cart exists
  - cart contains items
  - each cart product exists
  - quantities are positive
  - all products are active
  - stock is sufficient
- Shipping address validation:
  - `address`, `city`, `postal_code`, `country` required
- Payment validation:
  - `payment_method` is optional and defaults to Cash on Delivery
  - `payment_id` is required for non-COD methods

### Possible errors
- `400` cart empty
- `400` invalid quantity
- `400` product no longer exists
- `400` product inactive
- `400` insufficient stock
- `401` unauthorized
- `404` order not found

## 15. API Response Format

### Success response
- Most endpoints return plain JSON object(s)
- Example:
  {
    "id": 5,
    "name": "Wireless Keyboard",
    "price": "49.99"
  }

### Error response
- Standard error wrapper:
  {
    "success": false,
    "message": "Validation error.",
    "errors": { ... },
    "status_code": 400
  }

## 16. Validation Rules

### Password
- Must pass Django password validators
- Must match `password2` on registration

### Email
- Must be valid email format
- Must be unique

### Quantity
- Must be > 0
- Adding or updating cart items with zero or negative quantity triggers error / DB integrity failure

### Stock
- Checkout validates enough stock before order creation
- If stock is insufficient, checkout is rejected and cart remains intact

### Payment
- Requires `payment_method`
- Requires `payment_id`
- Amount is not accepted from the client

### Order status
- Admin can update `status`
- No strict backend enum for order status
- Common values:
  - `Pending`
  - `Shipped`
  - `Cancelled`

## 17. Error Codes

### 400
- Use for validation and malformed data
- Frontend should display user-friendly messages from `errors`
- Example:
  - invalid cart payload
  - insufficient stock

### 401
- Access token missing, invalid, or expired
- Frontend should redirect to login or refresh token
- `token/refresh` may be needed

### 403
- User lacks permission
- Frontend should show permission error or hide admin-only UI

### 404
- Resource not found
- Frontend should show a not-found page or toast

### 409
- Not explicitly used in backend
- Treat as business conflict if encountered

### 422
- Not explicitly used in backend
- No backend endpoint returns 422 by design

### 429
- Not currently configured in backend
- Not expected, but implement retry/backoff if it occurs

### 500
- Backend failure
- Frontend should show generic error page and log issue
